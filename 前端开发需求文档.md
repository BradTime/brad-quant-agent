# 量化投资 Agent 平台 - 前端开发需求文档

## 1. 项目概述

### 1.1 项目背景
量化投资 Agent 平台是一个基于人工智能的量化投资决策支持系统，旨在为投资者提供智能化的投资策略分析、实时市场监控、风险评估和交易建议等功能。

### 1.2 项目目标
- 提供直观、高效的用户界面，支持用户进行量化投资策略的创建、管理和监控
- 实现实时数据可视化展示，包括市场行情、策略回测结果、收益分析等
- 构建响应式设计，支持多设备访问
- 确保系统安全性和用户体验的平衡

### 1.3 技术栈
- **框架**: Next.js 15.x (App Router)
- **UI 库**: React 19.x
- **组件库**: shadcn/ui
- **语言**: TypeScript 5.x
- **样式**: Tailwind CSS 4.x
- **状态管理**: Zustand 4.x+（全局状态）
- **服务器状态**: @tanstack/react-query（API 数据管理）
- **数据可视化**: ECharts 5.x+

## 3. 非功能需求

### 3.1 性能要求
- **页面加载时间**: 首屏加载时间 < 2秒
- **交互响应时间**: 用户操作响应时间 < 200ms
- **数据加载**: 列表数据分页加载，支持虚拟滚动
- **图表渲染**: 大数据量图表采用数据采样，确保流畅渲染
- **图片优化**: 使用 Next.js Image 组件进行图片优化

#### 3.1.1 性能优化策略

**代码分割**:
- 使用 Next.js 动态导入（`dynamic import`）实现路由级别的代码分割
- 大组件和第三方库使用动态加载（如图表库、代码编辑器）
- 使用 `React.lazy` 和 `Suspense` 进行组件懒加载

**数据加载优化**:
- 列表数据使用分页（每页 20-50 条）
- 使用虚拟滚动（`react-window` 或 `react-virtualized`）处理长列表
- 实现无限滚动或"加载更多"机制
- 使用 React Query 的缓存和预取机制减少重复请求
- 实现数据预加载（鼠标悬停时预取详情数据）

**图表性能优化**:
- 大数据量时使用数据采样算法（LTTB 算法或 Douglas-Peucker 算法）
- 实现图表虚拟渲染，仅渲染可视区域
- 使用 Web Worker 处理数据计算（如技术指标计算）
- 图表交互操作使用防抖和节流

**缓存策略**:
- 静态资源使用浏览器缓存（设置合适的 Cache-Control）
- API 数据使用 React Query 的内存缓存
- 本地存储缓存用户偏好设置
- 使用 Service Worker 实现离线缓存（PWA）

**移动端性能优化**:
- 减少移动端初始加载数据量（仅加载关键数据）
- 使用图片懒加载和响应式图片
- 优化移动端交互（使用 CSS `will-change` 提示浏览器优化）
- 避免在移动端使用重量级动画

**性能监控指标**:
- **Core Web Vitals**:
  - LCP (Largest Contentful Paint) < 2.5s
  - FID (First Input Delay) < 100ms
  - CLS (Cumulative Layout Shift) < 0.1
- **自定义指标**:
  - API 响应时间 < 500ms（P95）
  - 图表渲染时间 < 100ms
  - WebSocket 消息延迟 < 50ms

### 3.2 兼容性要求
- **浏览器支持**:
  - Chrome/Edge (最新 2 个版本)
  - Firefox (最新 2 个版本)
  - Safari (最新 2 个版本)
- **设备支持**:
  - 桌面端 (1920x1080, 1366x768)
  - 平板端 (768px - 1024px)
  - 移动端 (375px - 768px)

### 3.3 可访问性要求
- 遵循 WCAG 2.1 AA 标准
- 支持键盘导航
- 提供适当的 ARIA 标签
- 支持屏幕阅读器
- 颜色对比度符合标准

### 3.4 安全要求
- 所有 API 请求使用 HTTPS
- 敏感数据加密传输
- XSS 防护
- CSRF 防护
- 输入验证和过滤
- 定期安全审计

### 3.5 可维护性要求
- 代码规范统一（ESLint + Prettier）
- 组件化开发，可复用性高
- 完善的代码注释
- TypeScript 类型定义完整
- 单元测试覆盖率 > 70%

#### 3.5.1 错误处理与监控

**错误边界（Error Boundaries）**:
- 在路由级别和关键组件添加错误边界
- 错误边界捕获渲染错误，显示友好的错误页面
- 记录错误信息并上报到监控系统（Sentry）

**错误处理策略**:
- API 错误：统一在请求拦截器中处理，根据错误码显示相应提示
- 网络错误：显示网络异常提示，提供重试按钮
- 业务错误：显示具体业务错误信息（如"策略参数无效）
- 未预期错误：显示通用错误提示，记录详细错误日志

**错误监控**:
- 集成 Sentry 进行错误追踪
- 记录错误堆栈、用户信息、操作路径
- 设置错误告警（错误率超过阈值时通知）
- 定期分析错误报告，优化错误处理逻辑

**用户体验优化**:
- 所有错误提示使用友好的文案，避免技术术语
- 提供错误恢复机制（如重试、刷新、返回首页）
- 关键操作失败时自动保存用户输入数据

## 4. UI/UX 设计规范

### 4.1 设计风格
- **整体风格**: 现代化、简洁、专业
- **色彩系统**:
  - 主色调: 深蓝色 (#1e3a8a) - 代表专业、可信
  - 辅助色: 绿色 (#10b981) - 收益/上涨
  - 辅助色: 红色 (#ef4444) - 亏损/下跌
  - 中性色: 灰色系用于背景和文字
- **字体系统**:
  - 中文字体: 思源黑体 / PingFang SC / Microsoft YaHei
  - 英文字体: Inter / -apple-system
  - 代码字体: JetBrains Mono / Monaco
- **间距系统**: 4px 基础单位，使用 Tailwind 默认间距
- **圆角系统**: 统一使用 4px、8px、12px

### 4.2 组件库要求
- 建立统一的组件库
- 组件包括但不限于:
  - Button（按钮）
  - Input（输入框）
  - Select（下拉选择）
  - Table（表格）
  - Card（卡片）
  - Modal（弹窗）
  - Toast（提示）
  - Loading（加载）
  - Chart（图表容器）
  - Pagination（分页）
  - DatePicker（日期选择器）
  - Form（表单）
- 组件支持暗色模式
- 组件支持响应式设计

### 4.3 交互设计
- **加载状态**: 所有异步操作提供加载反馈
- **错误处理**: 友好的错误提示，避免技术术语
- **空状态**: 提供有意义的空状态插画和提示文字
- **确认操作**: 危险操作（删除、交易等）需要二次确认
- **表单验证**: 实时验证，错误提示清晰
- **快捷键**: 常用操作支持键盘快捷键

### 4.4 响应式设计
- **移动端优先**: 采用移动端优先的设计策略
- **断点设置**:
  - sm: 640px
  - md: 768px
  - lg: 1024px
  - xl: 1280px
  - 2xl: 1536px
- **布局适配**: 
  - 移动端: 单列布局，隐藏次要信息
  - 平板: 双列布局，优化间距
  - 桌面: 多列布局，充分利用空间

#### 4.4.1 移动端交互设计

**手势操作**:
- 左右滑动：切换 Tab 或视图（如策略列表切换视图）
- 下拉刷新：列表数据刷新
- 上拉加载：加载更多数据
- 双击缩放：图表缩放操作

**移动端优化**:
- 触摸目标大小至少 44x44px
- 减少移动端输入字段（使用选择器替代文本输入）
- 移动端隐藏复杂图表，使用简化版本或仅显示关键数据
- 移动端数据加载策略：初始仅加载最近 7 天数据，用户可手动扩展

**性能优化**:
- 移动端减少动画效果
- 使用 CSS `transform` 和 `opacity` 实现动画（GPU 加速）
- 移动端图片使用 WebP 格式
- 禁用移动端不必要的功能（如代码编辑器，改为仅查看模式）

## 5. 技术架构

### 5.1 项目结构
```
src/
├── app/                    # Next.js App Router 页面
│   ├── (auth)/             # 认证相关页面（路由组）
│   │   ├── login/
│   │   └── register/
│   ├── dashboard/          # 仪表盘
│   ├── strategies/         # 策略管理
│   ├── backtest/           # 回测分析
│   ├── monitor/            # 实时监控
│   ├── trading/            # 交易执行
│   ├── settings/           # 设置中心
│   └── layout.tsx          # 根布局
├── components/             # 通用组件
│   ├── ui/                 # 基础 UI 组件
│   ├── charts/             # 图表组件
│   ├── forms/              # 表单组件
│   └── layout/             # 布局组件
├── lib/                    # 工具函数
│   ├── api/                # API 调用
│   ├── utils/              # 工具函数
│   └── constants/          # 常量定义
├── stores/                 # 状态管理（Zustand）
├── hooks/                  # 自定义 Hooks
├── types/                  # TypeScript 类型定义
└── styles/                 # 全局样式
```

### 5.2 状态管理方案
- **全局状态**: 使用 Zustand（轻量级、易用）
  - 用户信息
  - 主题设置
  - 全局配置
- **服务器状态**: 使用 @tanstack/react-query（已确定）
  - API 数据缓存
  - 自动重新获取
  - 乐观更新
  - 请求去重和重试机制
- **表单状态**: React Hook Form
- **URL 状态**: Next.js 路由参数和查询字符串

### 5.3 API 设计
- **API 基础路径**: `/api/v1`
- **请求拦截**: 统一处理 token、错误、 loading
- **响应格式**: 统一响应格式
  ```typescript
  {
    code: number,
    message: string,
    data: T,
    timestamp: number
  }
  ```
- **错误处理**: 统一错误处理中间件
- **类型安全**: 使用 TypeScript 定义 API 类型

#### 5.3.1 主要 API 端点清单

**认证相关**:
- `POST /api/v1/auth/login` - 用户登录
- `POST /api/v1/auth/register` - 用户注册
- `POST /api/v1/auth/logout` - 用户登出
- `POST /api/v1/auth/refresh` - 刷新 Token
- `GET /api/v1/auth/me` - 获取当前用户信息

**策略管理**:
- `GET /api/v1/strategies` - 获取策略列表（支持分页、筛选、排序）
- `POST /api/v1/strategies` - 创建策略
- `GET /api/v1/strategies/:id` - 获取策略详情
- `PUT /api/v1/strategies/:id` - 更新策略
- `DELETE /api/v1/strategies/:id` - 删除策略
- `POST /api/v1/strategies/:id/enable` - 启用策略
- `POST /api/v1/strategies/:id/disable` - 停用策略
- `POST /api/v1/strategies/:id/duplicate` - 复制策略

**回测分析**:
- `POST /api/v1/backtest/run` - 执行回测
- `GET /api/v1/backtest/:id` - 获取回测结果
- `GET /api/v1/backtest/:id/metrics` - 获取回测指标
- `GET /api/v1/backtest/:id/report` - 获取回测报告
- `POST /api/v1/backtest/:id/export` - 导出回测报告

**实时监控**:
- `GET /api/v1/market/quotes` - 获取市场行情
- `GET /api/v1/market/kline` - 获取K线数据
- `GET /api/v1/positions` - 获取持仓列表
- `GET /api/v1/positions/:id` - 获取持仓详情

**交易执行**:
- `POST /api/v1/trading/orders` - 提交订单
- `GET /api/v1/trading/orders` - 获取订单列表
- `GET /api/v1/trading/orders/:id` - 获取订单详情
- `DELETE /api/v1/trading/orders/:id` - 撤销订单

**风险管理**:
- `GET /api/v1/risk/metrics` - 获取风险指标
- `GET /api/v1/risk/alerts` - 获取风险告警
- `POST /api/v1/risk/stop-loss` - 设置止损
- `POST /api/v1/risk/take-profit` - 设置止盈

**数据管理**:
- `GET /api/v1/data/sources` - 获取数据源列表
- `POST /api/v1/data/sources` - 添加数据源
- `PUT /api/v1/data/sources/:id` - 更新数据源
- `DELETE /api/v1/data/sources/:id` - 删除数据源
- `POST /api/v1/data/sources/:id/test` - 测试数据源连接
- `GET /api/v1/data/history` - 获取历史数据

**用户设置**:
- `GET /api/v1/user/profile` - 获取用户资料
- `PUT /api/v1/user/profile` - 更新用户资料
- `POST /api/v1/user/avatar` - 上传头像
- `PUT /api/v1/user/password` - 修改密码

- `PUT /api/v1/user/settings` - 更新用户设置

#### 5.3.2 错误码定义

| 错误码 | HTTP 状态码 | 说明 | 处理建议 |
|--------|-----------|------|---------|
| 200 | 200 | 成功 | - |
| 400 | 400 | 请求参数错误 | 检查请求参数 |
| 401 | 401 | 未授权/Token 失效 | 跳转登录页 |
| 403 | 403 | 无权限访问 | 提示权限不足 |
| 404 | 404 | 资源不存在 | 提示资源不存在 |
| 409 | 409 | 资源冲突 | 提示冲突原因 |
| 422 | 422 | 业务逻辑错误 | 显示业务错误信息 |
| 429 | 429 | 请求过于频繁 | 提示稍后重试 |
| 500 | 500 | 服务器内部错误 | 提示系统错误，记录日志 |
| 503 | 503 | 服务不可用 | 提示服务维护中 |

**业务错误码**（code 字段）:
- `10001-10099`: 认证相关错误
- `10101-10199`: 策略管理相关错误
- `10201-10299`: 回测相关错误
- `10301-10399`: 交易相关错误
- `10401-10499`: 风险管理相关错误

### 5.4 WebSocket 实时通信
- **连接管理**: 
  - 自动重连机制（指数退避策略，最多重试 5 次）
  - 心跳检测（每 30 秒发送 ping）
  - 连接状态显示（连接中、已连接、断开、重连中）
  - Token 认证（连接时携带认证信息）

#### 5.4.1 WebSocket 消息协议

**连接地址**: `wss://api.example.com/ws/v1?token={token}`

**消息格式**:
```typescript
// 客户端发送消息格式
{
  type: string,           // 消息类型
  payload?: any,          // 消息负载
  requestId?: string      // 请求ID（用于请求-响应匹配）
}

// 服务端推送消息格式
{
  type: string,           // 消息类型
  payload: any,           // 消息负载
  timestamp: number,      // 时间戳
  requestId?: string      // 请求ID（响应消息）
}
```

**消息类型定义**:

**客户端 → 服务端**:
- `subscribe` - 订阅主题
  ```typescript
  {
    type: 'subscribe',
    payload: {
      topics: ['market.quote', 'strategy.signal']  // 订阅的主题列表
    }
  }
  ```
- `unsubscribe` - 取消订阅
  ```typescript
  {
    type: 'unsubscribe',
    payload: {
      topics: ['market.quote']
    }
  }
  ```
- `ping` - 心跳检测
  ```typescript
  {
    type: 'ping'
  }
  ```

**服务端 → 客户端**:
- `pong` - 心跳响应
  ```typescript
  {
    type: 'pong',
    timestamp: 1234567890
  }
  ```
- `market.update` - 市场行情更新
  ```typescript
  {
    type: 'market.update',
    payload: {
      symbol: 'AAPL',
      price: 150.25,
      change: 2.5,
      changePercent: 1.69,
      volume: 1000000,
      timestamp: 1234567890
    }
  }
  ```
- `strategy.signal` - 策略信号
  ```typescript
  {
    type: 'strategy.signal',
    payload: {
      strategyId: 'strategy-123',
      signal: 'buy' | 'sell' | 'hold',
      symbol: 'AAPL',
      price: 150.25,
      confidence: 0.85,
      timestamp: 1234567890
    }
  }
  ```
- `trade.executed` - 交易执行
  ```typescript
  {
    type: 'trade.executed',
    payload: {
      orderId: 'order-123',
      strategyId: 'strategy-123',
      symbol: 'AAPL',
      side: 'buy' | 'sell',
      quantity: 100,
      price: 150.25,
      status: 'filled',
      timestamp: 1234567890
    }
  }
  ```
- `risk.alert` - 风险告警
  ```typescript
  {
    type: 'risk.alert',
    payload: {
      alertType: 'stop_loss' | 'take_profit' | 'max_drawdown',
      strategyId: 'strategy-123',
      message: '触发止损',
      level: 'warning' | 'critical',
      timestamp: 1234567890
    }
  }
  ```
- `connection.error` - 连接错误
  ```typescript
  {
    type: 'connection.error',
    payload: {
      code: number,
      message: string
    }
  }
  ```

**订阅主题列表**:
- `market.quote.{symbol}` - 单个股票行情（如 `market.quote.AAPL`）
- `market.quote.*` - 所有行情更新
- `strategy.{strategyId}.signal` - 策略信号
- `strategy.*.signal` - 所有策略信号
- `position.{positionId}.update` - 持仓更新
- `trade.order.{orderId}` - 订单状态更新
- `risk.alert` - 风险告警

## 6. 开发规范

### 6.1 代码规范
- **命名规范**:
  - 组件: PascalCase (例: `StrategyCard.tsx`)
  - 函数/变量: camelCase (例: `getStrategyList`)
  - 常量: UPPER_SNAKE_CASE (例: `API_BASE_URL`)
  - 文件: kebab-case (例: `strategy-list.ts`)
- **代码格式化**: Prettier（配置文件已存在）
- **代码检查**: ESLint（配置文件已存在）
- **Git 提交**: 遵循 Conventional Commits 规范
  - feat: 新功能
  - fix: 修复 bug
  - docs: 文档更新
  - style: 代码格式调整
  - refactor: 重构
  - test: 测试相关
  - chore: 构建/工具链相关

### 6.2 组件开发规范
- **函数式组件**: 优先使用函数式组件和 Hooks
- **组件拆分**: 单个组件不超过 300 行
- **Props 类型**: 使用 TypeScript 严格定义 Props 类型
- **组件文档**: 关键组件添加 JSDoc 注释
- **性能优化**: 
  - 使用 `React.memo` 优化不必要的重渲染
  - 使用 `useMemo` 和 `useCallback` 优化计算和函数

### 6.3 文件组织规范
- **同目录原则**: 相关文件放在同一目录
- **Barrel Export**: 使用 `index.ts` 统一导出
- **路径别名**: 使用 `@/` 作为 `src/` 的别名

### 6.4 测试要求
- **单元测试**: 使用 Vitest
- **组件测试**: 使用 React Testing Library
- **E2E 测试**: 使用 Playwright（关键流程）
- **测试覆盖率**: 
  - 整体覆盖率 > 70%
  - 关键功能模块（交易执行、策略管理、风险管理）覆盖率 > 85%
  - 工具函数覆盖率 > 90%

#### 6.4.1 测试策略

**单元测试**:
- 覆盖所有工具函数和业务逻辑函数
- Mock 外部依赖（API、WebSocket）

**组件测试**:
- 测试组件渲染和基本交互

## 8. 国际化支持

### 8.1 支持语言
- 简体中文（默认）
- English（待实现）

### 8.2 国际化方案
- **方案选择**: 使用 `next-intl`（推荐，与 Next.js App Router 兼容性更好）
- **文本内容管理**: 
  - 所有文本内容提取到 `messages/` 目录
  - 按模块组织语言文件（如 `messages/zh-CN/dashboard.json`）
  - 使用命名空间管理（如 `dashboard.title`, `strategy.name`）
- **本地化格式化**:
  - 日期格式化：使用 `date-fns` 的 `format` 和 `formatDistance`
  - 数字格式化：使用 `Intl.NumberFormat`
  - 货币格式化：使用 `Intl.NumberFormat` 的货币格式
  - 时区处理：统一使用 UTC，前端根据用户设置转换
- **实现步骤**:
  1. 安装并配置 `next-intl`
  2. 创建语言文件结构
  3. 提取现有文本到语言文件
  4. 在组件中使用 `useTranslations` Hook
  5. 配置路由和中间件支持语言切换
